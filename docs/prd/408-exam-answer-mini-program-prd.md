## 1. 产品概述

### 1.1 产品背景

考研 408 统考包含数据结构、计算机组成原理、操作系统和计算机网络 4 门课程。备考用户通常需要大量刷题、反复复盘错题，并通过章节和知识点维度定位薄弱环节。

目前常见刷题方式存在以下问题：

- **题目分散：** 真题、章节题和解析分布在不同资料中，复习路径不连续。
- **错题复盘低效：** 用户需要手动记录错题，难以按科目、章节和知识点回看。
- **综合题反馈慢：** 主观题需要人工对照答案，自评不稳定，难以及时获得改进建议。
- **移动端使用不便：** 零散时间刷题需要轻量、快速、低负担的移动端工具。

### 1.2 产品目标

构建一个面向考研 408 用户的刷题训练小程序，支持按科目、章节和知识点刷题，提交后查看解析，自动记录错题、收藏和进度，并通过 AI 对综合题进行自动判分。

首版目标是跑通完整闭环：

```text
录入题目 → 用户刷题 → 提交答案 → 自动判分 → 查看解析 → 记录错题/收藏 → 统计进度
```

### 1.3 产品定位

首版产品定位为 **刷题训练型小程序**。

不优先做完整模拟考试、社区讨论、会员体系和复杂 AI 助教，而是先解决用户最核心的刷题、判分、解析和复盘需求。

### 1.4 目标用户

- **408 备考学生：** 需要系统刷题和复盘知识点。
- **二战或在职备考用户：** 时间碎片化，需要移动端快速练习。
- **强化阶段用户：** 需要按章节查漏补缺。
- **冲刺阶段用户：** 需要集中复盘错题和收藏题。

### 1.5 成功标准

首版上线成功需要满足以下标准：

- 用户可以在微信小程序中完成章节刷题流程。
- 管理员可以通过 Web 管理后台录入、编辑和管理题目。
- 单选题和多选题可以自动判分。
- 综合题可以通过 AI 返回得分、点评和改进建议。
- 错题、收藏和刷题进度可以按用户 OpenID 持久化保存。
- 小程序可通过微信开发者工具完成预览、真机调试、上传和审核。

以下功能不进入 MVP，作为后续版本规划：

- 用户社区和评论区

### 2.3 范围边界

首版以「可上线、可刷题、可管理、可复盘」为核心，不追求一次性覆盖所有 408 题库和复杂学习系统。

题库内容首版允许手动录入少量高质量题目，用于跑通产品闭环。后续再扩展题库规模和导入能力。

## 4. 用户故事

### 4.1 章节刷题

作为一名正在复习数据结构的考生，我希望可以选择「数据结构 → 树与二叉树」章节进行刷题，这样我可以针对刚学完的内容及时练习。

**验收标准：**

- 用户可以从首页进入科目列表。
- 用户可以选择科目和章节。
- 系统展示该章节下可练习题目。
- 用户可以连续答题并查看解析。

### 4.2 错题复盘

作为一名冲刺阶段的考生，我希望系统自动记录我做错的题目，这样我可以在考前集中复盘薄弱点。

**验收标准：**

- 用户答错单选题或多选题后，系统自动加入错题本。
- 综合题 AI 判分低于设定阈值时，系统自动加入错题本。
- 用户可以在错题本中再次练习。
- 用户答对错题后，系统可以更新错题状态。

### 4.3 收藏重点题

作为一名备考用户，我希望可以收藏典型题或易混题，这样我可以在复习时快速找到它们。

**验收标准：**

- 题目详情页提供收藏入口。
- 用户可以收藏或取消收藏。
- 收藏状态需要持久化保存。
- 收藏夹可以展示全部收藏题目。

### 4.4 综合题 AI 判分

作为一名练习综合题的考生，我希望提交主观答案后获得 AI 判分和点评，这样我可以知道自己的答案缺少哪些关键点。

**验收标准：**

- 用户可以在综合题答题页输入文本答案。
- 提交后系统调用云函数进行 AI 判分。
- 判分结果包含得分、点评、命中要点、遗漏要点和改进建议。
- AI 判分失败时，系统提供清晰提示，并允许用户稍后重试或查看参考答案。

### 4.5 管理员录题

作为管理员，我希望可以通过 Web 管理后台录入题目，这样我可以持续扩充题库而不需要直接改数据库。

**验收标准：**

- 管理员可以登录后台。
- 管理员可以新增单选题、多选题和综合题。
- 管理员可以维护题目分类、答案、解析和难度。
- 录入后的题目可以在小程序端被查询和展示。

## 6. 功能详述

### 6.1 首页

首页用于承载用户进入刷题流程的主入口，并展示基础学习状态。

#### 输入

- 用户 OpenID
- 用户刷题统计
- 科目和章节进度

#### 处理逻辑

- 首次进入时调用登录云函数获取 OpenID。
- 查询用户学习统计。
- 查询 4 门科目的基础进度。
- 如果用户有未完成练习，展示「继续刷题」入口。

#### 输出

- 今日刷题数量
- 总刷题数量
- 正确率
- 错题数量
- 科目入口
- 继续刷题入口

### 6.2 科目与章节选择

科目与章节选择用于帮助用户按复习范围进入练习。

#### 输入

- 科目 ID
- 章节 ID
- 知识点 ID（可选）

#### 处理逻辑

- 查询科目下章节列表。
- 查询章节下题目数量和用户完成情况。
- 支持按题型和知识点筛选。

#### 输出

- 章节名称
- 章节题量
- 已完成数量
- 正确率
- 开始练习入口

### 6.3 答题页

答题页是小程序核心页面，支持单选题、多选题和综合题。

#### 单选题

- 用户只能选择 1 个选项。
- 提交后与标准答案严格匹配。
- 答对计入正确记录。
- 答错计入错题记录。

#### 多选题

- 用户可以选择多个选项。
- 首版采用严格匹配规则：选项集合完全一致才算正确。
- 错选、漏选、多选均判为错误。

#### 综合题

- 用户通过文本框输入答案。
- 提交后调用 `gradeSubjectiveAnswer` 云函数。
- 云函数返回 AI 判分结果。
- AI 判分结果不作为绝对权威答案，需要在页面提示「AI 判分仅供参考」。

### 6.4 解析页

解析页用于展示答题结果和学习反馈。

#### 客观题解析

- 展示用户答案。
- 展示正确答案。
- 展示是否答对。
- 展示文字解析。
- 展示关联知识点。

#### 综合题解析

- 展示 AI 得分。
- 展示评分点评。
- 展示命中要点。
- 展示遗漏要点。
- 展示参考答案。
- 展示改进建议。

### 6.5 错题本

错题本用于自动沉淀用户薄弱题目。

#### 进入错题本的规则

- 单选题答错。
- 多选题答错。
- 综合题得分低于题目满分的 60%。

#### 移出错题本的规则

首版不自动删除错题，而是更新状态：

- `unresolved`：未掌握。
- `reviewing`：复习中。
- `resolved`：已掌握。

用户可以手动标记为已掌握。

### 6.6 收藏夹

收藏夹用于保存用户主动标记的重要题目。

#### 处理逻辑

- 用户点击收藏后写入 `favorites` 集合。
- 用户取消收藏后更新收藏状态或删除记录。
- 收藏状态需要在题目详情和列表中同步展示。

### 6.7 学习统计

学习统计用于给用户提供复习反馈。

#### 首版统计指标

- 总刷题数
- 今日刷题数
- 总正确率
- 各科目刷题数量
- 各科目正确率
- 错题数量
- 收藏数量

#### 后续可扩展指标

- 连续学习天数
- 知识点掌握度
- 周/月趋势图
- 高频错题知识点

### 6.8 管理后台题目管理

管理后台用于题库维护，首版仅管理员本人使用。

#### 新增题目字段

- 题型
- 科目
- 章节
- 知识点
- 题干
- 选项
- 正确答案
- 解析
- 难度
- 年份
- 来源
- 综合题评分规则
- 状态

#### 题目状态

- `draft`：草稿。
- `published`：已发布。
- `disabled`：已下架。

## 7. 题型规则

### 7.1 单选题

| 项目 | 规则 |
|------|------|
| 选项数量 | 通常为 4 个，支持扩展。 |
| 正确答案 | 1 个。 |
| 判分方式 | 用户答案与标准答案一致即正确。 |
| 得分 | 正确得满分，错误得 0 分。 |

### 7.2 多选题

| 项目 | 规则 |
|------|------|
| 选项数量 | 通常为 4 到 6 个，支持扩展。 |
| 正确答案 | 多个。 |
| 判分方式 | 严格匹配标准答案集合。 |
| 得分 | 完全正确得满分，否则得 0 分。 |

### 7.3 综合题

| 项目 | 规则 |
|------|------|
| 作答方式 | 文本输入。 |
| 判分方式 | AI 按参考答案和评分规则判分。 |
| 输出结果 | 得分、点评、命中要点、遗漏要点、改进建议。 |
| 兜底策略 | AI 失败时展示参考答案，并允许重试。 |

## 9. AI 判分需求

### 9.1 基本原则

- 小程序端不直接调用 AI API。
- AI API Key 只存放在微信云函数环境变量中。
- 云函数通过模型适配层调用具体 AI 服务。
- 首版默认接入 DeepSeek。
- 后续可扩展通义千问、阿里云百炼、智谱 GLM 或其他模型。

### 9.2 判分输入

综合题判分需要以下输入：

- 题干
- 用户答案
- 参考答案
- 评分规则
- 满分
- 科目
- 章节
- 知识点

### 9.3 判分输出

AI 判分结果必须转换为统一结构：

```json
{
  "score": 8,
  "maxScore": 10,
  "hitPoints": ["命中的关键点"],
  "missingPoints": ["遗漏的关键点"],
  "comment": "总体点评",
  "suggestion": "改进建议"
}
```

### 9.4 模型适配层

模型适配层负责屏蔽不同模型 API 的差异。

```text
小程序端
  ↓
微信云函数 gradeSubjectiveAnswer
  ↓
AI Grading Service
  ↓
Model Adapter
  ├─ DeepSeek Adapter
  ├─ Qwen Adapter（后续）
  └─ GLM Adapter（后续）
```

### 9.5 错误兜底

AI 判分失败时，需要返回明确状态：

- `AI_TIMEOUT`：模型响应超时。
- `AI_PROVIDER_ERROR`：模型服务异常。
- `AI_RESPONSE_PARSE_ERROR`：模型返回无法解析。
- `AI_QUOTA_EXCEEDED`：额度不足或限流。

页面兜底策略：

- 提示「AI 判分暂时不可用，请稍后重试」。
- 允许用户查看参考答案。
- 记录失败日志，便于后续排查。

## 10. 非功能需求

### 10.1 性能需求

- 首页首屏在正常网络下 2 秒内可用。
- 答题页切换题目无明显卡顿。
- 题目列表分页加载，避免一次性拉取大量数据。
- AI 判分需要显示加载状态，超时时间首版建议设置为 30 秒。

### 10.2 安全需求

- 用户数据必须按 OpenID 隔离。
- 小程序端不得保存 AI API Key。
- 管理后台接口必须校验管理员身份。
- 管理后台密码不得明文存储。
- 云函数需要校验请求来源和必要参数。
- 题目删除优先使用软删除或下架状态。

### 10.3 隐私与合规需求

- 小程序需要提供隐私政策说明。
- 需要说明收集用户 OpenID、昵称、头像和学习记录的用途。
- AI 判分页面需要提示「AI 判分仅供学习参考」。
- 不承诺 AI 判分与真实考试人工阅卷完全一致。

### 10.4 兼容性需求

- 首版目标平台为微信小程序。
- 代码采用 uni-app + Vue 3 + TypeScript，尽量通过 CLI 和源码工程在 Windsurf 中完成开发。
- 最终仍需使用微信开发者工具进行预览、真机调试、上传和审核。

### 10.5 可维护性需求

- 小程序端、云函数和管理后台应保持清晰边界。
- 题型判分逻辑应模块化，避免散落在页面中。
- AI 模型调用必须通过适配层，避免业务逻辑绑定单一模型。
- 题目数据结构需要支持后续导入和批量维护。

## 11. 验收标准

### 11.1 小程序端验收

- 用户可以完成微信登录并获取 OpenID。
- 用户可以从首页进入 4 门科目。
- 用户可以按科目和章节进入刷题。
- 单选题可以作答、提交和查看解析。
- 多选题可以作答、提交和查看解析。
- 综合题可以输入答案并获得 AI 判分。
- 用户答错题目后，错题本可以看到对应记录。
- 用户可以收藏和取消收藏题目。
- 用户可以查看基础学习统计。

### 11.2 管理后台验收

- 管理员可以登录后台。
- 管理员可以新增单选题、多选题和综合题。
- 管理员可以编辑题目。
- 管理员可以下架题目。
- 管理员可以维护科目、章节和知识点。
- 后台录入的已发布题目可以在小程序端展示。

### 11.3 云函数验收

- `login` 可以返回 OpenID 并创建用户记录。
- `getQuestionList` 可以按条件返回题目列表。
- `submitAnswer` 可以完成客观题判分。
- `gradeSubjectiveAnswer` 可以调用 AI 并返回统一判分结构。
- `toggleFavorite` 可以正确更新收藏状态。
- `getWrongQuestions` 可以返回用户错题列表。
- `getStudyStats` 可以返回基础学习统计。

## 12. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 题库质量不足 | 影响用户体验和可信度 | 首版手动录入少量高质量题目，优先保证闭环。 |
| AI 判分不稳定 | 综合题体验受影响 | 增加参考答案展示、重试机制和「仅供参考」提示。 |
| 微信审核不通过 | 无法上线 | 提前准备隐私政策，避免违规内容和敏感承诺。 |
| 云函数超时 | AI 判分失败 | 设置超时兜底，记录日志，必要时异步处理。 |
| 管理后台权限过弱 | 题库数据有风险 | 首版至少实现管理员登录和接口校验。 |
| 数据结构设计不合理 | 后续扩展困难 | 首版预留科目、章节、知识点、来源、年份和难度字段。 |

## 14. 待确认事项

以下事项不阻塞 PRD v1.0，但在进入详细实现计划前需要逐步确认：

- DeepSeek API 的具体模型和计费策略。
- 微信云开发环境 ID。
- 管理后台部署方式。
- 首批题库录入数量。

## 15. 结论

首版应优先构建「刷题训练闭环」，通过微信小程序承载用户刷题体验，通过 Web 管理后台维护题库，通过微信云开发保存用户数据和调用 AI 判分。

该方案可以在 Windsurf 中完成主要代码开发，并通过微信开发者工具完成最终预览、调试、上传和审核。
